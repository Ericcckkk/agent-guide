# Vibe Coding 工作流

AI 驱动开发的核心方法论和踩坑经验。适用于任何 agent 帮我从零开发新功能或新项目时。

---

## 核心理念

我的角色是导演，agent 的角色是执行者。我负责规划、决策和验收，agent 负责实现。

---

## 工作流（四步法）

- [ ] **Step 1 探索：先读后动。** 接到任务先读相关代码和文档，搞清楚现状，不要上来就改文件。
- [ ] **Step 2 规划：先出方案。** 把任务拆成可验证的小步骤列出来让我确认，不要直接动手写。
- [ ] **Step 3 执行：小步实现。** 每次只改一件事，改完确认再做下一个。不要一口气改 5 个文件。
- [ ] **Step 4 验证：跑通再提交。** 每步完成后跑测试或让我验证效果，通过了再 commit。

---

## Prompt / 上下文管理

- [ ] **上下文 > 提示词。** 与其写复杂 prompt，不如直接把相关文件/代码贴给 agent。提供上下文比雕琢提示词有效 10 倍。
- [ ] **需求要具体。** 不要说"帮我优化这个页面"，要说"把首页加载速度从 3s 降到 1s，主要是图片懒加载和接口合并"。
- [ ] **一次只提一个需求。** 不要在一条消息里堆 5 个功能。做完一个确认了再说下一个。
- [ ] **给约束条件。** 告诉 agent 技术栈限制、不能动的文件、性能预期等边界条件。

---

## 版本控制策略

- [ ] **每完成一个小功能就 commit。** commit message 要有意义，方便回滚时找到正确的点。
- [ ] **改崩了直接回滚，不要在错误代码上打补丁。** 先让 agent 分析错在哪，记住结论，回滚到好的版本，重新来。
- [ ] **用功能分支。** 大改动开分支，验证通过再合并到 main。
- [ ] **commit 前确认能跑。** 不要提交编译不过或跑不起来的代码。

---

## 模型选择

- [ ] **简单任务用轻量模型。** 改个文案、格式化代码、写注释，不需要最贵的模型。
- [ ] **复杂架构设计用强模型。** 系统设计、多文件重构、复杂 bug 排查，用最强的模型。
- [ ] **不确定时先问再做。** 如果 agent 对需求有歧义，宁可问一句确认，不要猜着做完再推倒重来。

---

## 文档驱动开发

- [ ] **大功能先写需求文档。** 超过 3 步的任务，先让 agent 出一份 Markdown 格式的方案（做什么、怎么做、验收标准），我确认后再动手。
- [ ] **规则文件必须维护。** 每个项目都应该有一份 agent 能读的规则文件（CLAUDE.md / .cursorrules / AGENTS.md），定义代码规范和架构约束。
- [ ] **DEVLOG.md 是活文档。** 每次工作完更新进度，下次任何 agent 接手都能直接看状态（这条已在 projects.md 中要求）。

---

## 踩坑经验

- [ ] **AI 生成的代码必须审查安全性。** 特别是输入校验、SQL 注入、CORS 配置等。个人项目 CORS 全开没事，但不要给公网 API 无脑全开权限。
- [ ] **不要让 AI 凭记忆写 API 调用。** 给它官方文档或示例代码作为上下文，否则它会编造不存在的参数和方法。
- [ ] **大型重构前先备份。** git stash 或新建分支，确保有退路。
- [ ] **AI 倾向于过度工程化。** 我要的是最简方案（见 general.md），如果 agent 给了过度抽象的设计，直接打回要求简化。
- [ ] **上下文窗口有限。** 大项目不要一次性把所有文件塞给 agent，只给当前任务相关的文件。精确指定路径。
- [ ] **迭代而非一次成型。** 第一版出个 60 分的，看哪里不对精确反馈，3-5 轮迭代比指望一次完美高效得多。
- [ ] **测试不能省。** 特别是重构后。"看起来能跑"和"没有回归 bug"是两回事。

---

## 新项目启动清单

开始一个新项目时，agent 应按以下顺序操作：

1. 确认技术栈（参考 coding-style.md 和 frontend.md 的偏好）
2. 创建 GitHub 仓库 + 基础目录结构
3. 写 README.md（架构、命令、端口）
4. 写 DEVLOG.md（初始状态）
5. 配置 CI/CD（参考 deployment.md）
6. 开始开发第一个功能（小步走，完成即 commit）
