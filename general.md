# 通用协作原则

## 沟通方式

- [ ] **给完整方案，不要让我一步步试。** 不要每次只给一个命令让我执行看结果再给下一个。一次性给出完整的解决方案。
- [ ] **出错时先想清楚再回复。** 不要急着给一个可能错的答案，宁可多想 10 秒给一个对的。
- [ ] **不要重复问已经确认过的事。** 如果我说过"不用备案了"，后面就别再提备案的事。
- [ ] **操作前先搞清楚环境。** 比如端口被谁占了、目录在哪、进程是谁的，查清楚再动手，不要盲目操作别人的服务。
- [ ] **认真读上下文。** 对话里已经说过的信息不要再问一遍。

## 工作风格

- [ ] **我要一劳永逸的方案。** 不要临时 workaround，要能长期稳定运行的方案。
- [ ] **效率优先。** 能自动化的就自动化，能一条命令搞定的不要分三条。
- [ ] **代码要放 GitHub。** 方便我在任何电脑维护，push 即部署。
- [ ] **中间文件用完即删。** 生成脚本、临时文件完成任务后立即清理。

## 禁止事项

- [ ] **不要动别人的端口/进程。** 先确认端口归属，不是自己的不要碰。
- [ ] **不要给半成品。** 如果方案有多步，全部写完再给我，不要写一半让我先试。
- [ ] **不要过度解释。** 我问怎么做，直接给操作步骤，不需要大段背景知识。

## Git 同步纪律

- [ ] **动手前先 git pull。** 开始任何项目工作前，先拉取最新代码，确保不会基于过时的版本开发。
- [ ] **做完后 push + 更新 DEVLOG。** 工作完成后，更新 DEVLOG.md 并 commit + push，让其他设备/agent 能看到最新状态。
- [ ] **冲突时以服务器 DEVLOG 为准。** 如果本地和远程有分歧，SSH 到服务器 cat DEVLOG.md 确认真实状态。
